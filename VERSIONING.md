# Versioning & Releases

How GHDS packages are versioned, what counts as a breaking change, and how a
release actually happens. This governs both contributors (which bump to pick)
and consumers (what a version bump means for you).

## Semantic Versioning

GHDS follows [SemVer](https://semver.org/): `MAJOR.MINOR.PATCH`.

- **MAJOR** — a breaking change (see [what "breaking" means](#what-counts-as-breaking) below).
- **MINOR** — new, backward-compatible functionality.
- **PATCH** — backward-compatible bug fixes.

### Pre-1.0 caveat (read this)

While a package is on `0.x`, **breaking changes may ship in a MINOR release**.
This is standard SemVer for pre-1.0 software and it happens in practice — for
example, `@ghds/react` `0.3.0` (a *minor*) renamed the low-level export
`buildRectangleOutline(...)` to `buildOutline(...)`, which was breaking for
anyone importing it directly. Pin exact versions if you depend on unstable
surfaces, and always read the changelog / [`MIGRATIONS.md`](./MIGRATIONS.md)
before bumping a `0.x` dependency.

Once a package reaches `1.0.0`, breaking changes require a MAJOR bump and a
migration guide.

## What counts as breaking

"Breaking" is defined per package type, because tokens and components fail
differently.

### Tokens (`@ghds/tokens`)

**Breaking:**

- Removing or renaming a token.
- Changing a token's `$type`.
- A value change that alters visual output beyond a reasonable tolerance
  (e.g. a hue shift a consumer has visually calibrated against), or that
  changes WCAG contrast outcomes.

**Minor:** adding a new token (any tier).

**Patch:** an imperceptible value correction, or metadata/description fixes.

### Components (`@ghds/react`, `@ghds/web-components`, `@ghds/react-native`)

**Breaking:**

- Removing or renaming a prop / attribute / exported symbol.
- Changing a prop's default in a way that changes rendered behavior.
- Removing a CSS custom-property hook or a documented DOM/ARIA contract.

**Minor:** a new optional prop, a new variant, a new component.

**Patch:** bug fixes that don't change the documented API or behavior.

Also note the **experimental** lifecycle stage
([`GOVERNANCE.md`](./GOVERNANCE.md#component-lifecycle)): an experimental
component may take breaking changes in a minor regardless of the package's
`1.0` status.

## Independent versioning

Packages are versioned **independently** (Changesets config has no `fixed` or
`linked` group). Each package bumps according to its own changesets. In
practice they often move together because a single change touches `tokens` plus
one or more libraries — but that is emergent, not enforced.

`updateInternalDependencies` is set to `patch`: when an internal `@ghds/*`
dependency bumps, its dependents get at least a patch bump so the dependency
range stays current.

## How a release happens

Releases are automated and gated. The pipeline
([`.github/workflows/release.yml`](./.github/workflows/release.yml)):

1. **Author changesets** during normal development (`pnpm changeset`). Each PR
   that touches a published package carries its changeset.
2. **CI gate.** The Release workflow runs via `workflow_run` **only after the
   `CI` workflow succeeds on `main`** — a red `main` never cuts a release.
3. **Version PR.** When unreleased changesets exist, the Changesets action opens
   or updates a **"Version Packages"** PR that bumps versions and regenerates
   each package's `CHANGELOG.md`.
4. **Publish.** When that PR is merged (no changesets remain), the workflow runs
   `pnpm run release` (`turbo build && changeset publish`) and publishes to npm
   via **OIDC trusted publishing** — no `NPM_TOKEN`, and provenance is attached
   automatically. Only the maintainer merges the Version Packages PR.

### Known limitation: CI does not run on the Version Packages PR

The "Version Packages" PR is authored with the default `GITHUB_TOKEN`, and
GitHub **does not trigger workflows for token-authored events**. So the standard
`CI` checks do **not** run on that PR itself. This is expected. The safety comes
from step 2: the release is gated on CI passing on `main` *before* the version
PR is cut, so the versioned content has already been validated. If you want CI
to also run on the version PR, supply a PAT or GitHub App token as
`GITHUB_TOKEN` in the release workflow.

## Changelogs

Each published package has its own `CHANGELOG.md`, generated by Changesets from
changeset entries. Entries cross-reference the Linear issue (`GHD-##`). There is
no aggregated root changelog; read the per-package files under
`packages/*/CHANGELOG.md`.
