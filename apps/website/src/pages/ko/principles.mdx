디자인 스타일, 컴포넌트, 패턴은 GHDS가 어떻게 보이고 그 조각들이 어떻게 맞물리는지를
설명해요. 이 페이지는 그 이면의 이유, 즉 모든 토큰과 컴포넌트 결정 뒤의 원칙, 그리고
GHDS가 함께 제공하는 모든 단어 뒤의 보이스 & 톤을 설명해요.

## 디자인 원칙

다섯 가지 원칙이 오늘날 GHDS를 이끌어요. 이 집합은 의도적으로 작으며, 시스템이 성숙해지면
일곱 개로 늘어날 수도 있어요 — 하지만 아래의 모든 원칙은 이미 코드베이스에서 실제 결정을
이끌고 있어요.

### 1. 인간적이되 신뢰할 수 있게

**정의.** 손그림 스타일은 미완성이나 아마추어처럼 보이지 않으면서 따뜻함을 더해요.

**배경.** 스케치 렌더링은 의도적인 정체성 선택이지 자리표시자 미학이 아니에요 — 고위험
순간을 포함한 실제 제품 사용에서도 견뎌야 해요.

**하세요**
- 스케치 채우기와 손그림 스트로크를 일러스트나 장식 표면(카드, 버튼, 빈 상태 아트)에
  사용하세요.
- 숫자 값, 법적 텍스트, 폼 입력은 정확하고 스케치가 아닌 타이포와 아이콘으로 유지하세요.

**하지 마세요**
- 정확함이 신뢰로 읽히는 안전 중요 확인 대화상자나 조밀한 데이터 표에 거칠기를
  적용하지 마세요.

> 전체 배경과 거칠기 토큰 가이드는 <a href={withBase('/about/#why-hand-drawn')}>소개 & 접근성 → 왜 손그림인가?</a>를 참고하세요.

### 2. 토큰이 진실이고, 예외는 문서화돼요

**정의.** 모든 디자인 값은 `@ghds/tokens`로 추적돼요; 어떤 이탈도 문서화되고 의도적인
예외예요.

**배경.** 단일 진실 공급원이 세 플랫폼(React, Web Components, React Native)의 시각적 일관성을
유지하는 핵심이에요 — 조용한 드리프트는 그 보장을 깨뜨려요.

**하세요**
- 모든 색상, 간격, 반경, 스케치 파라미터에 대해 `sys.*` 또는 `comp.*` 토큰을 참조하세요.
- 문서화된 일회성 변형을 위해 도입된 새 `sys` 티어 역할에는 설명 주석을 추가하세요.

**하지 마세요**
- hex 값, 픽셀 값, 숫자 거칠기를 하드코딩하지 마세요 — 또는 `comp` 토큰이 `ref`를 직접
  참조하게 하여 `comp → sys → ref` 체인을 조용히 깨뜨리지 마세요.

### 3. 접근성이 기본이에요

**정의.** 접근 가능한 동작은 옵션이 아니라 자동으로 제공돼요.

**배경.** 접근성을 나중에 덧붙이는 건 비싸고 불안정해요; 모든 컴포넌트의 기본 상태에
구축하는 것은 그렇지 않아요.

**하세요**
- 모든 컴포넌트에 키보드 포커스 상태, `aria-*` 속성, 의미 있는 라벨을 기본으로 제공하세요.
- 빌드 시점에 WCAG 2.1 AA 기준으로 색상/배경 페어링을 검증하세요.

**하지 마세요**
- 접근성을 컨슈머가 활성화하라고 기억해야 하는 prop으로 취급하지 마세요.

### 4. 플랫폼 중립적이고 일관된 경험

**정의.** React, Web Components, React Native 모두 동일한 토큰을 소비하고 동일한 경험을
제공해요.

**배경.** 팀이 디자인을 웹에서 네이티브로 옮길 때 다시 배우거나 다시 승인할 필요가 없어야
해요 — 플랫폼은 구현 디테일이에요.

**하세요**
- 세 패키지 모두에서 동일한 `sys`/`comp` 토큰 세트로 모든 컴포넌트를 구축하세요.
- 컴포넌트를 출시하기 전에 플랫폼 간 시각적 동일성을 확인하세요.

**하지 마세요**
- 한 플랫폼의 구현이 플랫폼별 색상, 간격, 동작으로 조용히 달라지게 하지 마세요.

### 5. 장식보다 명확함

**정의.** 손그림 텍스처는 따뜻함을 더하지만, 가독성이나 위계를 희생하지 않아요.

**배경.** 개성은 수단이지 목적이 아니에요 — 빠르게 읽을 수 없는 시스템은 본래 역할에
실패한 거예요.

**하세요**
- 스케치 텍스처로 표면을 부드럽고 인간적으로 만들되, 타이포, 대비, 간격이 위계의 실질적인
  역할을 하도록 유지하세요.

**하지 마세요**
- 장식적 거칠기가 가독성을 떨어뜨리거나, 상태를 가리거나, 화면을 스캔하는 데 드는 노력을
  늘리게 하지 마세요.

## 보이스 & 톤

### 보이스

GHDS의 글쓰기는 다음과 같아요:

- **따뜻해요** — 귀엽게가 아니에요. 열정을 연기하지 않고 친근해요.
- **직접적이에요** — 무뚝뚝하지 않아요. 일어난 일과 다음에 할 일을 군더더기 없이 말해요.
- **정직해요** — 자극적이지 않아요. 문제를 과장 없이 평이하게 말해요.
- **차분해요** — 무미건조하지 않아요. 오류와 성공 모두에서 안정적이에요.

### 상황별 톤

| 상황 | 톤 조정 |
| --- | --- |
| 성공 | 간결하고 사실적인 확인 — 추가적인 열정 없이. |
| 오류 | 차분하고 구체적이며 실행 가능하게. 무슨 일이 일어났는지와 다음에 할 일을 설명하고; 절대 사용자를 탓하지 마세요. |
| 빈 상태 | 격려하고 부담을 낮추며, 하나의 명확한 다음 행동을 제시. |
| 온보딩 | 친근하고 자신감을 심어주며, 전문 용어 없이. |

### 이렇게 하지 마세요

- 과장이나 과대 광고 언어("Amazing!", "You're crushing it!").
- 위협적이거나 비난하는 오류 문구("You broke the form.", "Fatal error.").
- 신뢰를 떨어뜨리는 억지 친근함 — [원칙 1](#1-human-but-trustworthy) 참고.
- 인간적인 보이스를 깨는 기업적이거나 법률적인 딱딱함.
- 비슷한 상태에서 일관성 없는 톤 — 예: 장난스러운 온보딩과 로봇 같은 오류의 조합.

구체적인 오류 메시지 템플릿, 버튼 라벨 규칙 등 상세 마이크로카피 규칙은 UX 라이팅
가이드(작업 중)에서 다뤄요.
