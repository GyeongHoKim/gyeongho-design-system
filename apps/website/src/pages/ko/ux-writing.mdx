## 개요

이 가이드는 GHDS의 <a href={withBase('/principles/#voice--tone')}>보이스 &amp; 톤</a>을
인터페이스의 단어 — 버튼 라벨, 오류 메시지, 빈 상태, 확인 — 에 대한 구체적이고 재사용
가능한 규칙으로 바꿔요. 보이스 &amp; 톤은 *GHDS가 어떻게 들리는지*를 말해요; 이 가이드는
특정 화면에 도달했을 때 *무엇을 입력해야 하는지*를 말해요.

디자이너, PM, 엔지니어 모두를 위해 작성됐어요: 버튼에 이름을 붙이거나 오류 문구를 쓸 때,
기능마다 새로 발명하기보다는 여기서 답을 찾을 수 있어야 해요.

## 보이스, 한 줄 요약

GHDS의 글쓰기는 따뜻하고, 직접적이고, 정직하고, 차분해요 — 열정을 연기하지 않고 친근하며,
무뚝뚝하지 않고 평이해요. 상황별 톤 표를 포함한 전체 정의는
<a href={withBase('/principles/#voice--tone')}>디자인 원칙 → 보이스 &amp; 톤</a>에 있으며
여기서 반복하지 않아요; 그 페이지가 단일 진실 공급원이에요. 아래는 그 네 단어가 실제로
어떻게 구현되는지를 보여줘요.

## 버튼 & 링크

버튼 라벨은 눌렀을 때 일어날 일에 대한 약속이에요. **명령문 동사구**로, 문장형 대소문자로,
끝에 문장 부호 없이 작성해요. 일반적인 "OK"나 "Submit"보다 구체적인 동작 동사("Delete",
"Save changes", "Send invite")를 선호해요 — 라벨은 주변 텍스트를 가려도 올바르게
읽혀야 해요.

목적지가 탐색일 때는 **링크**를, 결과가 동작일 때는 **버튼**을 사용해요. 링크 텍스트는
목적지나 결과("View billing history")를 나타내지, 방식("Click here")을 나타내지 않아요.

  ]}

## 오류 메시지

오류 메시지의 역할은 하나예요: 사용자가 앞으로 나아가도록 돕는 것. **무슨 일이
일어났는지**와 **다음에 할 일**을 평이한 언어로, 사용자를 탓하지 않고 말해요. 시스템
전문 용어("null", "exception", "invalid input")와 오류 코드는 주요 메시지로 사용하지
마세요 — 코드는 지원을 위해 작은 텍스트로 뒤따를 수 있지만, 헤드라인이 되어서는 안 돼요.

톤은 차분하고 구체적으로 유지해요. 다음 단계 없이 "Something went wrong"은 오류 메시지가
아니에요; 막다른 길이에요.

<DoDont
  do={[
    '일어난 일과 할 일을 말하세요: "That email is already registered. Try signing in instead."',
    '해결책을 가리키세요: "Add at least one recipient before sending."',
    '인간적이고 구체적으로 유지하세요 — 필드나 초과된 한도의 이름을 지정하세요.',
    '오류를 관련 필드에 연결하고, 나타날 때 (role="alert") 알리세요.',
  ]}
  dont={[
    '사용자를 탓하지 마세요: "You entered an invalid value."',
    '내부를 노출하지 마세요: "Error: NullPointerException at line 42."',
    '"Something went wrong."과 다음 단계 없이 막다른 길을 만들지 마세요.',
    '과장하지 마세요 — "Fatal error" 없이, 느낌표 없이.',
  ]}
/>

## 빈 상태

빈 상태는 실패가 아니라 시작이에요. 한 문장으로 사용자를 안내한 다음, **하나의 명확한
다음 행동**을 제공해요. 부담을 낮추고 톤을 격려적으로 유지해요 — 이것은 누군가의 기능 첫
경험인 경우가 많아요.

<DoDont
  do={[
    '여기에 무엇이 나타날지와 시작하는 방법을 설명하세요: "No projects yet. Create one to get going."',
    '위의 버튼 라벨 규칙에 맞는 단일 주요 행동을 제공하세요.',
    '빈 상태를 활용해 친근한 목소리로 기능을 간단히 알려주세요.',
  ]}
  dont={[
    '설명이나 행동 없이 빈 패널을 남기지 마세요.',
    '여러 경쟁하는 행동 유도를 쌓지 마세요.',
    '사과하거나 사용자가 잘못했다는 뉘앙스를 주지 마세요.',
  ]}
/>

## 성공 & 확인

성공은 **간결하고 사실적으로** 확인해요 — 말의 폭죽은 없어요. 무슨 일이 일어났는지
이름을 붙인 짧은 토스트나 인라인 메시지면 충분해요: "Changes saved", "Invite sent".

파괴적 동작 전의 **확인 대화상자**에서는, 제목이 구체적인 결과를 이름 붙이고 확인 버튼이
동사를 반복해야 해요. 누군가 버튼만 읽어도 선택이 모호하지 않도록요. 모호한 "Are you sure?"
는 피하세요 — 읽는 사람이 자신이 무엇을 하고 있었는지 떠올리게 만들어요.

<DoDont
  do={[
    '평이하게 확인하세요: "Changes saved" — 그리고 길을 비키세요.',
    '파괴적 대화상자는 결과로 제목을 지으세요: "Delete this project?"',
    '확인 버튼을 동작에 맞추세요: "Delete project" / "Cancel".',
    '동작을 되돌릴 수 없는 경우, 대화상자 본문에 한 번 알려주세요.',
  ]}
  dont={[
    '과도하게 축하하지 마세요: "Woohoo! You did it!"',
    '파괴적 대화상자를 "Are you sure?"로 "Yes" / "No" 버튼과 함께 표현하지 마세요.',
    '확인 버튼과 취소 버튼이 같아 보이거나 같게 읽히게 하지 마세요.',
  ]}
/>

## 용어집 / 용어

개념당 하나의 용어를 선택하고 어디에나 사용하세요 — UI, 문서, 오류 문구. 일관성 없는
동의어("remove"를 여기서, "delete"를 저기서)는 인터페이스를 덜 신뢰할 수 있게 만들어요.
이 표는 GHDS의 시작 어휘예요; 병렬 용어를 발명하기보다 제품별로 확장하세요.

| 개념 | 사용 | 피하기 |
| --- | --- | --- |
| 무언가를 영구히 제거 | **Delete** | Remove, Destroy, Purge |
| 목록에서 항목을 꺼냄 (복구 가능) | **Remove** | Delete |
| 인증된 세션에 진입 | **Sign in** | Log in, Login (verb) |
| 세션에서 나감 | **Sign out** | Log out, Logout (verb) |
| 계정 생성 | **Sign up** | Register, Join |
| 기존 데이터에 편집 저장 | **Save changes** | Update, Apply |
| 저장하지 않은 편집 폐기 | **Discard** | Cancel, Reset |
| 동작 없이 닫기 | **Cancel** | Dismiss, Nevermind |
| 필수 필드가 비어 있음 | **Required** | Mandatory, Must |

sign in / sign up 구분에 주의하세요: "sign in"은 이미 계정이 있는 사람을 위한 것이고,
"sign up"은 계정이 없는 사람을 위한 것이에요 — "login"을 동사로 사용하지 마세요 (명사예요,
"the login screen"처럼).

## 이중언어 원칙 (한국어 / 영어)

GHDS 제품은 한국어와 영어로 출시될 수 있어요. 규칙은 **번역을 나중에 덧붙이는 것이 아니라
동등성**이에요: 모든 사용자 대면 문자열은 두 언어로 존재하고, 개념적으로 병행하며, 둘 다
위의 기준을 충족해야 해요 (한국어 오류 메시지도 원인과 해결책을 명시해야 하고; 한국어
버튼도 동사구여야 해요).

**위의 영어 용어집이 정규**예요 — 새 용어는 영어로 먼저 결정된 다음, 고정된 한국어 대응어가
부여되어 번역자가 즉흥적으로 번역하는 대신 일대일로 매핑해요. 영어 용어당 하나의 한국어
용어를 유지하세요, 용어집이 개념당 하나의 영어 용어를 유지하는 것과 정확히 같아요.

| 영어 | 한국어 |
| --- | --- |
| Delete project | 프로젝트 삭제 |
| That email is already registered. Try signing in instead. | 이미 가입된 이메일이에요. 대신 로그인해 보세요. |
| No projects yet. Create one to get going. | 아직 프로젝트가 없어요. 하나 만들어 시작해 보세요. |

> 이 문서 사이트는 영어로 작성돼요 (<a href={withBase('/about/')}>소개</a>의 언어 규칙
> 참고). 위의 쌍은 제품 팀을 위한 KO/EN 동등성을 보여주는 예시예요 — 이 사이트가
> 이중언어라는 의미는 아니에요.

## 대소문자, 문장 부호 & 숫자

- **인터페이스 텍스트는 문장형 대소문자** — 버튼, 라벨, 메뉴 항목, 제품 내부의 제목. 첫
  단어와 고유명사만 대문자로 시작하세요; 실제 제품이나 브랜드 이름에만 타이틀 케이스를
  쓰세요. (이 가이드의 페이지 제목은 문서 사이트의 규칙을 따르며, 제품 UI와는 별개의
  맥락이에요.)
- **끝 문장 부호 없음** — 버튼, 라벨, 짧은 독립 메시지에는 끝문장 부호를 붙이지 마세요.
  여러 문장의 본문이나 완전한 문장 도우미 텍스트에는 마침표를 사용하세요.
- **숫자는 숫자로** (`3 items`, `Step 2 of 4`), 작은 숫자도 포함하여, 스캔 가능성을 위해.
- **축약형은 환영해요** ("you're", "we'll", "can't") — 보이스를 따뜻하게 유지해요. 다만
  귀여움으로 넘어가지 않도록 주의하세요.
- **마침표 뒤에는 공백 하나**, 곧은 아포스트로피와 따옴표를 작성된 대로 사용, 그리고
  말줄임에는 세 개의 점이 아닌 말줄임표 문자(…)를 사용하세요.
