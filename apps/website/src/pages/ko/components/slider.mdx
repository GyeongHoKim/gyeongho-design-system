## 개요

`Slider`는 트랙을 따라 썸을 드래그해서 연속적이거나 계단식 범위에서 숫자 값을 선택해요. 빠른 시각적
조정보다 정확한 타이핑 값이 더 중요할 때는 `Input type="number"`를 대신 사용하세요.

## 구조

스케치 스타일의 레일(얇은 사각형)과 채워진 부분(시작부터 썸까지), 그리고 둥근 썸 — 세 가지 도형 모두
`@ghds/sketch-core`에서 직접 계산되며, 하나의 측정된 박스와 시드를 공유해요. React와 Web Components에서는
보이지 않는 실제 `<input type="range">`가 전체 트랙을 차지하며 모든 인터랙션(드래그, 트랙 클릭, 네이티브
화살표/Home/End/PageUp/PageDown 키보드 동작)을 커스텀 포인터/keydown 코드 없이 처리해요. React Native에는
이런 네이티브 기본 요소가 없어서, 드래그를 `PanResponder`로 직접 구현했어요 (React Native 코어에 내장되어 있어요 —
새로운 제스처 라이브러리 의존성이 없어요).

## 접근성

- **React / Web Components**: 보이지 않는 실제 `<input type="range">`가 암시적 `role="slider"`와
  브라우저가 계산하는 `aria-valuenow`/`aria-valuemin`/`aria-valuemax`를 무료로 제공해요 —
  커스텀 ARIA 연결이 필요 없어요. `aria-valuetext`는 커스텀 음성 값을 위해 일반 속성으로 전달돼요.
- **React Native**: `accessibilityRole="adjustable"`와 `accessibilityValue={{min, max, now}}`,
  그리고 `accessibilityActions`/`onAccessibilityAction`(`increment`/`decrement`)이 VoiceOver/TalkBack의
  표준 RN 패턴이에요. `accessibilityValue`와 `accessibilityState.disabled` 둘 다 React Native Web에
  해당하는 매핑이 없어요 — `createDOMProps`에서 직접 확인했어요. 그래서 웹 타겟을 위해
  `aria-valuemin`/`aria-valuemax`/`aria-valuenow`/`aria-disabled`를 직접 설정해요. Select의
  `aria-expanded`와 Switch의 `aria-checked`에 이미 문서화된 것과 같은 차이예요. `onAccessibilityAction`
  자체는 React Native Web 연결이 전혀 없어요 (직접 확인함) — 네이티브 전용(iOS/Android) 기능이에요.
  자동화된 테스트가 아니라 `packages/react-native/AGENTS.md`의 디바이스/시뮬레이터 요구사항에 따라
  수동으로 검증했어요. `focusable`/`tabIndex`가 웹 타겟에서 컨트롤을 실제 키보드 탭 정지점으로 만들어요
  (`role="slider"`는 React Native Web의 자동 포커스 가능 역할 중 하나가 아니에요). 하지만 포커스 중
  화살표 키 단계 조정은 v1에서 아직 연결되지 않았어요 — `onKeyDown`은 실제 React Native의 `View` prop이
  아니라서 가벼운 추가가 아니에요. 마우스/터치로 드래그는 모든 플랫폼에서 작동해요.
- 일반적인 규칙은 <a href={withBase('/accessibility/')}>접근성 가이드</a>를 참고하세요.

## 콘텐츠

라벨은 조정 중인 값의 이름이에요 ("Volume", "Rating"), 현재 읽을 값이나 지시문이 아니에요.
