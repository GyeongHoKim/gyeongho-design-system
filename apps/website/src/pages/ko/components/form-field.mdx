## 개요

`FormField`는 단일 폼 컨트롤 주위에 Label + HelperText + ErrorText를 구성하고, 플랫폼이 허용하는 곳에서 `aria-describedby`/`aria-invalid`를 자동으로 연결해요. 다가오는 M12 폼 패턴 마일스톤의 기반이에요 — 컨트롤에 라벨, 도움말 메시지, 또는 오류가 필요할 때마다 화면마다 id/ARIA 연결을 직접 만드는 대신 이걸 사용하세요.

**React에서는** 래핑된 `Input`/`Textarea`에 `label`/`error`를 직접 전달하지 마세요 — `FormField`가 그것들의 렌더링을 담당해요; 래핑된 컨트롤은 context에서 id와 ARIA 속성을 읽어요.

## 구조

손그림 박스는 없어요 — `FormField`는 순수 레이아웃과 타이포그래피예요. 다음 순서로 렌더해요: 선택적 라벨, 래핑된 컨트롤(React/React Native에서는 `children`, Web Components에서는 `<slot>`), 선택적 도움말 텍스트 줄, 선택적 오류 줄. `helperText`와 `error`가 모두 제공되면 **둘 다 동시에 렌더**돼요 — 오류 시 숨김 전환이 없어요.

## 접근성

이것이 가장 중요한 섹션이에요 — 연결 메커니즘이 플랫폼마다 진짜로 달라요:

- **React**: `FormField`는 context로 `{ id, describedByIds, invalid }`를 제공해요. `Input`과 `Textarea`는 존재할 때 이를 읽고 자체 `id`/`aria-invalid`/`aria-describedby`를 계산해요 — 하지만 `FormField`가 제공하지 않은 경우 래핑된 컨트롤 자체의 `label`/`error`가 여전히 렌더돼요, 따라서 둘 다 전달해도 필드의 라벨이나 오류가 조용히 사라지지 않아요. Checkbox/Radio/Switch/Select/Slider는 아직 이 context를 소비하지 않아요 — 자체 오류 개념이 없어서 감싸면 라벨/도움말/오류 텍스트는 렌더하지만 아무것도 연결하지 않아요(그 컴포넌트들이 자체 오류 개념을 얻는 향후 더 큰 작업에서 다뤄져요).
- **Web Components**: `<gh-form-field>`는 `id`/`aria-describedby`/`aria-invalid`를 슬롯된 컨트롤에 직접 자동 연결해요 — 문자열 형태의 `aria-describedby`는 light-DOM 형제에서 `gh-form-field` 자신의 섀도 루트 안의 요소를 참조할 수 없어요(ID 참조는 섀도 경계를 넘지 않아요), 따라서 자신의 light-DOM 자식에 직접 접근해서 명령형으로 속성을 설정해요(호스트 자체 스크립트에서의 일반적인 무제한 DOM 접근 — 스타일 캡슐화와 슬롯 *프로젝션*만 섀도 범위예요). 마운트 후 래핑된 컨트롤이 교체되면 `slotchange`에서도 재동기화돼요.
- **React Native**: 이 플랫폼에는 id/`aria-describedby`에 해당하는 것이 아예 없어요. `FormField`는 시각적 구성만 해요 — 도움말/오류 텍스트와 래핑된 컨트롤 간의 자동 접근성 연결이 없어요. 컨트롤에 포커스하는 스크린 리더 사용자는 도움말/오류 텍스트를 자동으로 듣지 못해요. 이는 `accessibilityHint` 복제로 흉내 낸 것이 아니라 실제 영구적인 플랫폼 간극이에요.
- 일반적인 관례는 <a href={withBase('/accessibility/')}>접근성 가이드</a>를 참고하세요.

## 콘텐츠

라벨은 수집하는 값을 나타내요("Email"), 지시가 아니에요. 도움말 텍스트는 사용자가 실수하기 전에 형식이나 제약을 설명해요("Letters, numbers, and underscores only"); 오류 텍스트는 검증 후 무엇이 잘못되었고 어떻게 고치는지 설명해요.
