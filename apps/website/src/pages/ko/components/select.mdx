## 개요

`Select`는 버튼 트리거에 앵커된 플로팅 패널에 목록을 표시하여, 사용자가 하나의 옵션을 선택하게 해줘요.
다른 모든 GHDS 폼 컨트롤과 달리, 실제 네이티브 `<select>`를 감싸지 않아요 — 네이티브 드롭다운의
브라우저가 그리는 UI는 손그림 스타일에 맞게 다시 스타일링할 수 없기 때문이에요. 그래서 트리거는 실제
`<button>`이지만, 패널과 키보드 모델은 WAI-ARIA "select-only combobox" 패턴을 따라 직접 구현했어요.

## 구조

스케치 스타일의 트리거 버튼(선택된 값 또는 플레이스홀더, 열렸을 때 뒤집히는 셰브론 포함): 클릭하거나
Enter/Space/화살표 키를 누르면 트리거 아래(또는 공간이 없으면 위)에 위치한 플로팅 리스트박스 패널이 열려요.
실제 DOM 포커스는 트리거를 떠나지 않아요 — 현재 하이라이트된 옵션은 패널로 포커스를 옮기는 게 아니라
`aria-activedescendant`로 추적해요.

## 접근성

- **React / Web Components**: 트리거에 `role="combobox"` + `aria-expanded` + `aria-haspopup="listbox"` +
  `aria-controls`; 패널에 `role="listbox"`; 트리거의 `aria-activedescendant`가 하이라이트된 옵션의 id를
  가리켜요. 완전한 키보드 지원: 화살표 위/아래로 하이라이트 이동 (닫혀 있으면 패널을 먼저 열어요),
  Home/End로 첫 번째/마지막 활성화된 옵션으로 이동, Enter/Space로 선택, Escape로 닫기(포커스는 트리거에 유지),
  타이핑하면 방금 입력한 텍스트로 시작하는 첫 번째 옵션으로 이동하는 타입어헤드.
- **React Native**: `accessibilityRole="combobox"` + 직접 `aria-expanded` prop (React Native Web은
  `aria-disabled`처럼 `accessibilityState`에서 `aria-expanded`를 파생하지 않아요 — `react-native-web`
  소스코드에서 직접 확인했어요. `Checkbox`/`Radio`/`Switch`의 `aria-checked`에 이미 문서화된 것과 같은 차이예요).
  **`aria-haspopup`, `aria-controls`, `aria-activedescendant`는 React Native에 해당하는 것이 아예 없어요** —
  어떤 형태로든 지원되는 RN prop이 없기 때문에 수동 해결책도 존재하지 않아요. 옵션 패널은
  `accessibilityRole="menu"`/`"menuitem"`을 사용해요. 사용 가능한 가장 가까운 역할이에요 (RN에는
  `listbox`/`option` 역할이 없어요).
- 일반적인 규칙은 <a href={withBase('/accessibility/')}>접근성 가이드</a>를 참고하세요.

## 콘텐츠

플레이스홀더는 선택 대상의 이름을 나타내요 ("과일 선택"), 지시문이 아니에요 ("선택해 주세요" X).
옵션 라벨은 옵션 자체의 이름이지 동작이 아니에요.
